# Editorial Control Console Spec

更新日期：2026-04-03

這份文件定義下一階段的「站內控制台 / Notion 審查流 / 發布流」規格。

原則：
- 不碰使用者正式文案主體。
- 控制台處理的是流程、狀態、審查、發布，不是代替使用者撰寫內容。
- Notion 是編務資料後台，站內控制台是操作入口。

## 一、整體目標

下一階段希望做到：
- 在站內控制台處理文章與留言審查
- 在站內控制台追蹤 Inbox 訊息
- 清楚區分「待審核」、「已核准」、「已發布」、「已封存」
- 讓 Notion 作為資料底層，網站作為操作層

## 二、控制台頁面規劃

### 1. `/admin/review`
用途：
- 處理待審文章
- 處理待審留言

現況：
- 已有第一版可用頁面

下一階段補強：
- 篩選器
- 依狀態切換
- 顯示審查歷程摘要
- 更清楚的多管理者資訊

### 2. `/admin/articles`
用途：
- 查看所有文章
- 切換待審、已發布、退回修改、已封存
- 進入單篇管理頁

預期功能：
- 搜尋標題、作者、場次
- 依主要場次與引用來源場次篩選
- 顯示 likes / comments / 狀態

### 3. `/admin/articles/[id]`
用途：
- 單篇文章管理

預期功能：
- 顯示文章完整內容
- 顯示主要場次與引用來源場次
- 顯示聯絡信箱（不公開）
- 顯示審查備註
- 操作：
  - 核准
  - 退回
  - 刪除
  - 封存
  - 標記精選

### 4. `/admin/inbox`
用途：
- 管理 `Inbox_DB`

預期功能：
- 依類型查看：
  - 一般聯絡
  - 錯誤回報
  - 內容更正
  - 使用建議
  - 私密傳訊
- 顯示附件連結
- 標記處理狀態
- 內部備註

### 5. `/admin/logs`
用途：
- 查閱 `Moderation_Log_DB`

預期功能：
- 依 `target_type` 篩選
- 依 `actor_name` 篩選
- 顯示狀態變更前後

## 三、Notion 審查流

### 文章流
1. 使用者從單場次或跨場次工作區投稿
2. 寫入 `Articles_DB`
3. 狀態初始為 `待審核`
4. 管理者在 `/admin/review` 或 `/admin/articles/[id]` 操作
5. 審查紀錄同步寫入 `Moderation_Log_DB`

建議狀態定義：
- `待審核`
- `審閱中`
- `退回修改`
- `已核准`
- `已發布`
- `已封存`

### 留言流
1. 使用者在文章頁留言
2. 寫入 `Comments_DB`
3. 狀態初始為 `待審核`
4. 管理者核准或退回
5. 審查紀錄同步寫入 `Moderation_Log_DB`

### Inbox 流
1. 使用者從 `/contact` 提交
2. 寫入 `Inbox_DB`
3. 狀態初始為 `新進`
4. 管理者改為：
   - `處理中`
   - `已回覆`
   - `已結案`
   - `已封存`

## 四、發布流

### 現行模式
- 文章狀態為 `已發布` 時，前台 `/forum` 可讀到
- 文章內頁 `/forum/[id]` 顯示已發布文章與核准留言

### 下一階段補強
- `/admin/articles` 提供更清楚的發布操作
- 發布前可檢查：
  - 是否有標題
  - 是否有主要場次
  - 是否有來源場次
  - 是否有內容
- 發布後同步更新：
  - `published_at`
  - `last_activity_at`

## 五、角色規劃

### owner
- 全部權限
- 可刪除
- 可封存
- 可處理管理者設定

### reviewer
- 可審查文章與留言
- 可處理 Inbox
- 不可刪除核心資料

- 新增 `/admin/inbox`

### 第二優先

- 新增 `/admin/logs`

### 第三優先

## 七、與目前主架構的關係

這份控制台規格與目前主架構一致：
- 前端本地資料負責逐字稿與引用體驗
- Notion 負責文章、留言、Inbox、審查紀錄
- 站內控制台負責操作與治理

不再把網站正式文案放進 CMS 後台管理。
