## 2. 專案目前的產品定義

此網站不是一般新聞站，也不是傳統論壇。

它是一個以「法庭逐字閱讀、觀庭共構、審核後公開發表」為核心的專業知識平台。

目前產品邏輯應被理解為四層：

1. 首頁層
   - 負責說明平台定位、呈現計畫背景、展示最新還原與熱門內容
2. 觀庭共構層
   - 各場次工作檯
   - 讓使用者邊看逐字稿邊撰寫論述，並能精準引用逐字內容
3. 公開文章層
   - 匯集通過審核的觀庭共構文章
   - 提供公開閱讀、按讚、留言與分享
4. 支援知識層
   - 提供先備知識、說明與平台使用脈絡

## 3. 核心產品判斷

### 3.1 首頁不是本次重構目標

首頁目前整體方向被視為可接受且應保留。

這代表：

- `計畫緣起` 保留在首頁
- 首頁既有大型區塊原則上不作結構性刪除
- 本輪重構不以「重新規劃首頁內容」為主

首頁可以後續再微調，但不是這次大修改的主戰場。

### 3.2 這次重構的真正核心

本次大修改只聚焦三個核心模組：

1. `觀庭共構筆記工作檯`
2. `公開文章／論壇匯集頁`
3. `熱門排行榜與互動機制`

## 5. 論壇的新定義

### 5.1 舊理解

舊版規劃中，論壇曾被視為一個較自由的獨立討論區：

- 有多種分類
- 有較自由的發文模式
- 與觀庭筆記雖有關聯，但仍像一個獨立論壇模組

### 5.2 最新定義

論壇現在不再是獨立主產品，而是「觀庭共構工作檯產出的公開文章層」。

因此論壇的最新性質是：

- 文章來源只來自各個觀庭共構工作檯
- 使用者不再直接把論壇視為獨立自由投稿區
- 只有經審核通過的觀庭共構文章才會進入論壇匯集頁

也就是說：

- 工作檯是內容生產場
- 論壇是通過審核後的公開展示場

### 5.3 保留的舊機制

論壇內頁仍保留：

- 按讚機制
- 留言板機制
- 文章公開連結

因此論壇不是被刪除，而是被重新定義。

## 6. 觀庭共構工作檯的新定義

### 6.1 核心目標

工作檯頁的目標不是單純顯示逐字稿，而是讓使用者：

- 左側閱讀逐字內容
- 右側撰寫論述
- 點擊逐字片段快速插入引用
- 在文章中保留可追溯來源

### 6.2 工作檯應有的核心互動

1. 使用者進入單一場次頁 `/sessions/[id]`
2. 左側顯示該場次逐字稿
3. 右側提供富文本或準富文本編輯區
4. 點擊左側某段逐字時：
   - 插入引用標記
   - 該標記需對應原文 line / transcript id
5. 已完成的文章可送審
6. 審核通過後進入論壇／公開文章層

### 6.3 引用機制的正式定義

引用的功能目標有三個：

1. 文章內顯示可辨識的引用標記
2. 滑鼠移上引用時可顯示原始逐字內容
3. 點擊引用時可跳回原文位置

這個機制是本網站的差異化核心，不應被視為附屬功能。

### 6.4 工作檯頁不再主打段落留言

舊版思路中，逐字段落可能有大量獨立留言互動。

新方向下，工作檯的主功能不是「段落留言牆」，而是：

- 引用
- 共構
- 論述撰寫
- 送審產文

段落留言若保留，也應是次要功能，不可壓過工作檯主流程。

---

## 7. 熱門排行機制的新定義

熱門排行可以保留，但資料來源與邏輯要跟新版文章體系對齊。

### 7.1 應保留的排行概念

- 熱門文章
- 文章按讚排行
- 文章留言互動排行

### 7.2 不應再依賴的舊觀念

- 把論壇當獨立自由討論區的熱門排行
- 把過去多種舊類型貼文混在一起排行

### 7.3 最新排行資料來源

排行榜的核心資料應主要來自：

- 已審核通過的觀庭共構文章
- 文章按讚數
- 文章留言數

若未來仍保留「熱門逐字」或「熱門觀點」，必須在文件中明確標示它是次級排行，而不是與文章排行混淆。

## 8. 資料層最新原則

### 8.1 冷資料與熱資料分流

本專案現在明確區分兩種資料：

#### 冷資料

指不會頻繁變動、以閱讀與工作檯支援為主的資料，例如：

- 場次清單
- 單一場次 metadata
- 逐字稿內容

這些資料應盡量朝本地化、靜態化或可預載方式靠攏，以降低對外部資料源的依賴。

#### 熱資料

指會持續新增、互動、審核與公開的資料，例如：

- 投稿文章
- 留言
- 按讚
- 公開文章清單

這些資料仍可透過 API 與後端資料庫維護。

### 8.2 Notion 的最新定位

Notion 不再適合承擔整個工作檯的即時冷資料讀取主流程。

它現在更適合承擔：

- CMS 型資料
- 審核流程
- 投稿文章儲存
- 留言與按讚所需資料
- 公開文章清單

### 8.3 舊有動態抓取逐字稿架構的地位

舊版直接從 Notion 動態抓取 sessions / transcripts 的方式，已被視為過渡或待淘汰方案。

它不是新版架構的目標狀態。

## 9. 頁面角色最終定義

### 9.1 首頁 `/`

角色：

- 平台主入口
- 展示平台定位、計畫緣起、先備知識入口、最新場次與熱門內容

本輪處理原則：

- 保留現況結構
- 不作大規模砍區塊或搬家

### 9.2 場次頁 `/sessions`

角色：

- 場次入口頁
- 導向工作檯

它的任務是幫助使用者找到：

- 哪些場次已公開
- 哪些場次可進入工作檯

### 9.3 工作檯 `/sessions/[id]`

角色：

- 真正的核心生產頁
- 閱讀逐字 + 撰寫論述 + 插入引用 + 送審

### 9.4 論壇列表 `/forum`

角色：

- 已審核通過文章的匯集頁

不是：

- 自由論壇首頁

### 9.5 論壇文章頁 `/forum/[id]`

角色：

- 公開文章閱讀頁
- 可留言、可按讚、可分享

### 9.6 排行頁 `/rankings`

角色：

- 與已發布文章、互動數據連動的排行頁

## 12. 重構優先順序

後續實作應依照以下順序：

1. 重新收斂工作檯資料來源與引用格式
2. 打通工作檯送審文章流程
3. 重構論壇列表，使其只顯示已審核工作檯文章
4. 重構文章內頁，保留留言與按讚
5. 將排行榜改接新版文章資料來源
6. 最後才處理其他延伸優化
