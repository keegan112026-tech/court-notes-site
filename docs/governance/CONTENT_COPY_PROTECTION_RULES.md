# 文案與命名保護規則

最後更新：2026-04-02

## 核心原則

本專案中，正式對外文案、命名、定義、敘事方式，僅由使用者本人決定。

工程重構、資料流調整、元件改寫、後端切換，不構成改寫原始文案的理由。

## 強制規則

1. 未經使用者明確授權，不得改寫既有頁面文案。
2. 未經使用者明確授權，不得擅自變更頁面命名、區塊名稱、功能稱呼。
3. 如為支援新功能而必須新增文案，只能新增，不得覆蓋既有原文。
4. 如需重構頁面，應優先保留、繼承、延續原始文案，而非以新的摘要或改寫取代。
5. 若發現歷史版本眾多，應以使用者明確指定的正式頁面或基準頁為準，不得自行混用舊稿。
6. 若頁面需要大幅結構重排，應先確認：
   - 哪些是可調整的框架
   - 哪些是不可改動的內容資產

## 可做與不可做

### 可做

- 調整元件結構
- 調整資料流
- 修正 API 與後端
- 修正互動邏輯
- 修正編碼、亂碼、版面錯位
- 在不改動原文意思與措辭的前提下，將原始文案重新掛回新架構

### 不可做

- 用新的敘述取代使用者原文
- 因工程方便而任意簡化原文
- 將開發過程中的暫稱直接覆蓋正式頁面命名
- 將不同歷史版本的文案混合拼貼成新版本

## 執行方式

後續任何頁面修復或回掛，應先確認：

1. 這個頁面的 canonical baseline 是哪一個來源。
2. 此來源是否為使用者明確認可版本。
3. 如無基準來源，不應自行重寫內容，只能先保留現況或請使用者指定。

## 目前已確認原則

- `/sessions` 頁面已有使用者明確指定的正式基準頁，不應任意改寫。
- 首頁中的「計畫緣起」與使用者偏好的敘事區塊，屬於高敏感內容資產，未經授權不得重寫。
