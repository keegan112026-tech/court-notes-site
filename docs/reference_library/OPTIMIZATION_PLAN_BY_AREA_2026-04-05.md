# 各區塊優化計畫

建立日期：2026-04-05  
用途：把目前網站各部位的後續優化方向整理成可執行計畫。  
原則：**只作規劃，不代表立即實作。**

## 0. 共同原則

1. 不改你未明示要改的正式文案。
2. 優先修「可用性、可信度、一致性」，再談炫技。
3. 所有優化都要先對照 production 與基準文件。
4. 能用既有技術棧完成的，優先不額外引入新框架。

## 1. 首頁 `/`

### 目前定位
- 總入口頁
- 負責導流到還原筆記、計畫緣起、論壇、聯絡與支持

### 目前主要問題
- Hero 與中段內容仍有些拼接感
- 熱門區與主敘事之間還能再更自然
- 公共品牌語言雖已收斂，但首頁的節奏還可以再更完整

### 優化目標
- 讓首頁更像完整入口，而不是多塊內容併排
- 保持高質感，但避免過度裝飾
- 強化第一屏判讀：這是什麼、要去哪、先做什麼

### 建議行動
1. 收 Hero 左右欄比例與留白
2. 把熱門入口再做成更像次級導覽，而不是內容卡縮圖
3. 強化首頁與 `/sessions` 的銜接段落
4. 補 production smoke markers，避免之後首頁退化沒發現

### 推薦參考
- `shadcn-ui/taxonomy`
- `magicuidesign/magicui`

## 2. 還原筆記總覽 `/sessions`

### 目前定位
- 核心工作主線頁
- 提供案件總覽、本站須知、公開筆記入口、跨場次工作檯入口、程序時間線

### 目前主要問題
- 上方雙欄雖已可用，但仍可再更穩
- 某些區塊的寬度、卡片密度、閱讀節奏還能再調
- 案件總覽、本站須知、公開筆記之間的層級還有優化空間

### 優化目標
- 把 `/sessions` 打造成最穩的正式主頁面之一
- 讓新場次持續加入時，不會再破壞版面

### 建議行動
1. 微調雙欄比例與卡片高度
2. 收 timeline 的閱讀密度與視覺標示
3. 再明確區分「案件背景」、「本站須知」、「公開筆記」、「工作檯」
4. 讓新增場次後的 UI 連動完全制度化

### 推薦參考
- `react-chrono`
- `react-vertical-timeline`

## 3. 單場工作檯 `/sessions/[id]`

### 目前定位
- 逐字稿閱讀與共構編輯的工作頁

### 目前主要問題
- 手機版已改善，但還可以再減少操作摩擦
- 桌面 / 手機切換策略可再更細
- 引用插入、預覽、跳轉這條鏈還能更順

### 優化目標
- 手機上能順讀、順切換、順引用
- 桌面維持雙欄工作效率

### 建議行動
1. 再看手機版是否要把 `逐字紀錄 / 編輯器` 切換做得更明顯
2. 補充引用狀態提示與成功回饋
3. 減少送審流程中的不確定感
4. 補更多工作檯的空狀態與失敗狀態

### 推薦參考
- `ueberdosis/tiptap`
- `bvaughn/react-resizable-panels`

## 4. 論壇 `/forum` 與文章頁 `/forum/[id]`

### 目前定位
- 公開文章層
- 承接審核後公開的共構文章、經驗分享與觀點

### 目前主要問題
- 首頁導入與論壇頁角色仍可再更清楚
- 文章列表與文章閱讀頁的品牌連續性還可更完整

### 優化目標
- 讓 `/forum` 更像「專業文章層」
- 避免與首頁熱門入口互相重複

### 建議行動
1. 收斂文章卡片的層級與摘要密度
2. 再確認文章頁頭部與返回資訊夠不夠清楚
3. 強化論壇與還原筆記之間的角色分工

### 推薦參考
- `magicuidesign/blog-template`
- `shadcn-ui/taxonomy`

## 5. 規範與知識頁 `/guide`、`/knowledge`

### 目前定位
- 說明網站怎麼讀、怎麼用、為什麼存在
- 建立背景知識，讓讀者有能力判讀內容

### 目前主要問題
- 長期來看，文件頁層級還能更像完整 knowledge system
- 章節導航與跨頁連動仍可更清楚

### 優化目標
- 讓這兩頁更像有組織的知識庫，而不是單頁說明

### 建議行動
1. 強化頁內目錄與章節錨點
2. 清楚區分「規範」與「知識背景」
3. 若未來內容量持續增加，再評估 docs-style layout

### 推薦參考
- `fuma-nama/fumadocs`
- `outline/outline`

## 6. 公共導覽 / 公共外殼

### 目前定位
- 讓正式頁維持一致品牌與可回到主線的能力

### 目前主要問題
- 雖然已經大幅改善，但公共 shell 仍不是完全 root-level 統一
- 某些頁面仍靠個別頁自行帶 header

### 優化目標
- 降低每個頁面各自長不同殼的風險

### 建議行動
1. 評估是否建立真正的 public shell
2. 把品牌、beta 提示、footer、metadata 進一步模組化
3. 減少頁面自行硬寫導覽的機會

### 推薦參考
- `shadcn-ui/ui`
- `radix-ui/primitives`

## 7. 資料層 / 投稿流程 / 安全性

### 目前定位
- 支撐 sessions、forum、contact、like、moderation 等流程

### 目前主要問題
- `local-data` 雖已進入 codegen 方向，但仍需長期驗證
- API 與後端提供者之間還有可再簡化的地方
- 投稿、聯絡、按讚等互動仍需持續做可信度與防濫用補強

### 優化目標
- 讓資料流程更穩、更可持續更新
- 減少手動步驟與假資料風險

### 建議行動
1. 讓 session 新增流程持續走 codegen + verify
2. 持續避免前台展示假熱度或假狀態
3. 中期再評估是否把部分 API 收斂成更安全的 action 模式

### 推薦參考
- `TheEdoRan/next-safe-action`
- `payloadcms/payload`（長期替代方向）

## 8. 優先順序建議

### 第一優先
1. 首頁收尾
2. `/sessions` 區塊節奏與雙欄穩定化
3. `/sessions/[id]` 手機體驗再精修

### 第二優先
1. `/forum` 與首頁熱門入口的角色再明確
2. `/guide` / `/knowledge` 結構化
3. 公共 shell 再制度化

### 第三優先
1. 資料層進一步收斂
2. 更高階的後端或 action 架構評估

## 最後結論

這個專案現在最適合的策略不是大翻修，而是：

- 先把已有的好基礎收穩
- 讓首頁、`/sessions`、工作檯三條主線更成熟
- 再把知識頁與論壇層級補齊

也就是說，接下來最值得投資的不是「加更多頁」，而是把**核心主線做得更完整、更可信、更有完成度**。
