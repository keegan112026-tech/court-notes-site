# 《匯集區發佈系統技術決策稿 v0.1》

日期：`2026-04-05`  
狀態：`draft / pending adoption`

## 0. 目的

本文件用來鎖定「匯集區」的核心技術決策，避免後續只改頁面文案與卡片外觀，卻沒有同步定義：

- 審核通過後的文章如何進入前端公開層
- 單場次、跨場次、專題文章如何被系統判斷與陳列
- 前端與後端各自負責什麼
- 真人測試要如何驗證整條鏈真的可運作

本文件優先處理技術邊界，再處理資料模型與頁面陳列。

---

## 1. 背景與現況

目前專案已具備以下基礎能力：

1. `單場筆記頁` 與 `跨場工作檯` 可送出投稿
2. 投稿會寫入後端 provider，並以 `pending` 狀態進入待審池
3. `審查台` 可核准、退回、刪除文章
4. `匯集區` 可讀取已公開文章列表
5. `單篇文章頁` 可讀取單篇文章與留言

但目前仍有以下問題：

1. 已公開文章仍主要依賴 API 直接讀取全文
2. `匯集區` 的定位、陳列與資料類型尚未正式建模
3. 審核通過後「如何變成前端穩定公開頁」沒有被正式定義
4. `slug`、文章類型、公開索引、版本快照等公開層欄位尚未補齊

---

## 2. 決策重心

### 本系統的核心技術問題

> 審核通過後，文章究竟如何進入前端公開層？

在此問題沒有被鎖定之前，以下設計都會懸空：

- `匯集區` 卡片分類
- `單篇文章頁` 的網址結構
- SEO 與效能策略
- 真人測試腳本

因此，本文件先鎖這個決策，再往下展開。

---

## 3. 已評估方案

### 方案 A：全部即時由後端 API 供應公開文章

做法：
- 審核通過後，文章維持存在於後端
- 前端的 `匯集區` 與 `單篇文章頁` 每次都由 API 抓取全文

優點：
- 實作最直接
- 不需另做發佈匯出

缺點：
- 每次公開閱覽都依賴後端
- SEO、速度與穩定性受限
- 公開內容缺乏「穩定版本」概念
- 與「審核後應成為前端獨立頁」的產品方向不一致

### 方案 B：審核通過即自動寫入前端靜態檔並觸發部署

做法：
- 後端核准文章後自動產生靜態內容檔
- 自動更新 repo 或內容庫並觸發部署

優點：
- 最接近最終理想狀態
- 公開層穩定，SEO 最佳

缺點：
- 需要額外解決內容寫檔、部署觸發、權限、失敗回復
- 一開始就做全自動，風險較高
- 不利於先進行真人測試與流程校正

### 方案 C：雙階段發佈制（本文件採用）

做法：
1. 投稿先進後端審核池
2. 審核通過後，狀態進入 `ready_to_publish`
3. 由管理端執行一次「發布匯出」
4. 產生公開用 snapshot，寫入前端內容庫
5. 再由前端部署成正式公開頁

優點：
- 後端與前端責任清楚
- 已公開內容有穩定版本
- 不需每次全文都依賴後端
- 適合先做真人測試，再逐步升級到更自動化方案

缺點：
- 多一個「發布匯出」步驟
- 管理流程稍微長一點

---

## 4. 正式技術決策

### 決策 1：採用「後端審核池 + 前端 published snapshot + 手動發布匯出」

本專案短期採用方案 C。

也就是：

- 後端只負責：
  - 投稿接收
  - 待審管理
  - 審核狀態
  - 原始內容保存
  - 審核紀錄

- 前端只負責：
  - 已公開文章索引
  - 已公開文章獨立頁
  - 首頁「目前已形成論述」摘要區
  - `匯集區` 分類陳列

### 決策 2：公開頁不直接依賴後端全文 API

已公開文章不以即時全文 API 作為長期正式來源。  
API 可保留作管理、比對、回寫或 fallback，但正式公開頁應以前端 published snapshot 為主。

### 決策 3：先做手動發布匯出，不做全自動部署

在真人測試與流程尚未跑順前，不採用：

- 核准即自動寫入 repo
- 核准即自動部署
- 核准即 ISR 直接公開

先保留人工控制點，降低風險。

## 5. 內容生命週期

### 後端審核池狀態

- `pending`
- `rejected`
- `ready_to_publish`
- `archived`

### 前端公開層狀態

- `published`
- `withdrawn`

### 建議流程

1. 投稿建立 `pending`
2. 管理員於 `審查台` 進行初審
3. 核准後不是直接公開，而是進入 `ready_to_publish`
4. 管理員執行發布匯出
5. 產生 published snapshot
6. 前端部署後，正式對外顯示為 `published`

---

## 6. 文章類型決策

### 正式分類

前台公開層使用三種文章類型：

1. `single-session`
   - 單場次筆記

2. `multi-session`
   - 跨場次整合

3. `thematic`
   - 專題 / 綜合論述

### 類型判斷規則

為避免投稿者誤判類型，類型不完全由投稿者自行選擇。

#### 自動判斷

- 從 `單場筆記頁` 投稿
  - 自動預設為 `single-session`

- 從 `跨場工作檯` 投稿
  - 自動預設為 `multi-session`

#### 管理員可調整

- `thematic`
  - 作為特殊類型
  - 只在審核或發布前由管理員指定

### 補充原則

前台分類邏輯與後台原始欄位可以分離。  
也就是，後端保存來源資訊，前台陳列再依規則映射成對讀者最清楚的類型。

## 7. `slug` 決策

### 不在投稿當下生成 `slug`

原因：
- 投稿標題可能還在調整
- 審核前內容不應直接對應正式公開網址

### 在「準備發布」階段生成 `slug`

流程：
1. 文章進入 `ready_to_publish`
2. 系統先提出一個建議 `slug`
3. 管理員可於發布前微調一次
4. 最終寫入 published snapshot

### 建議生成策略

1. 以標題生成可讀字串
2. 後綴固定短碼，保證唯一性

例如：

- `child-abuse-identification-a1b2`
- `cross-session-social-work-analysis-f4k9`

### fallback

若標題無法穩定生成：

- `article-20260405-a1b2`

### 原則

- `slug` 一旦公開後，不應輕易變更
- 若必須更改，需保留 redirect 或舊 ID 對應

## 8. 資料模型調整

### 後端審核池文章模型（草案）

在現有 `BackendArticle` 上，建議補充：

- `draftSource`
  - `single-session-editor` / `multi-session-workbench`

- `articleType`
  - `single-session` / `multi-session` / `thematic`

- `primarySessionId`
  - 主要場次

- `sourceSessionIds`
  - 來源場次清單

- `excerpt`
  - 短摘要

- `reviewStage`
  - `pending` / `ready_to_publish` / `rejected`

- `slugDraft`
  - 尚未公開前的建議 slug

### 前端 published snapshot 模型（草案）

公開層建議至少包含：

- `id`
- `slug`
- `title`
- `excerpt`
- `contentHtml`
- `articleType`
- `primarySessionId`
- `sourceSessionIds`
- `authorLabel`
- `publishedAt`
- `reviewedAt`
- `reviewedBy`
- `coverTone`
- `likesSnapshot`
- `commentsOpen`

## 9. 前後端合作模式

### 後端責任

- 保存原始投稿內容
- 管理投稿狀態
- 管理審核記錄
- 提供待審資料給 `審查台`
- 提供匯出來源給發布腳本

### 前端責任

- 讀取 published snapshot
- 生成 `匯集區` 清單
- 生成 `單篇文章頁`
- 顯示文章類型、主要場次、來源場次數
- 提供首頁摘要入口

### 發布匯出責任

建議新增一個受控的匯出腳本或內部工具，負責：

1. 讀取 `ready_to_publish` 文章
2. 清洗內容
3. 生成 `slug`
4. 生成公開摘要
5. 寫入前端內容庫
6. 標記該批次已匯出

## 10. 前端公開層建議儲存方式

短期建議：

- `data/published-articles/index.json`
- `data/published-articles/{slug}.json`

理由：
- 與目前專案的本地 JSON 內容方式較一致
- 易於真人測試
- 不需要立刻引入新的 MDX / CMS 流程

中期可再討論是否升級成：
- MDX
- 內容型 repo
- 自動化發佈流程

---

## 11. 匯集區頁面陳列決策

`匯集區` 不應再只是單純文章列表。  
建議固定分為三區：

1. 單場次筆記
2. 跨場次整合
3. 專題 / 綜合論述

每張卡片至少顯示：

- 類型
- 主要場次
- 來源場次數量
- 發布日期
- 標題
- 摘要

### 之後的文案與名稱方向

對外名稱以：

- `觀庭筆記匯集區`

為正式名稱。  
避免再使用容易誤解成自由討論區的「論壇交流」作為核心定位。

## 12. 真人測試計畫

### 第一輪：正常流程

1. 從 `單場筆記頁` 投稿一篇
2. 從 `跨場工作檯` 投稿一篇
3. 確認 `審查台` 看到兩筆待審
4. 核准一篇、退回一篇
5. 執行發布匯出
6. 確認核准的文章出現在 `匯集區`
7. 確認 `單篇文章頁` 可開啟
8. 確認退回文章不出現在公開頁

### 第二輪：類型與陳列

1. 單場次文章
2. 跨場次文章
3. 專題文章
4. 確認 `匯集區` 分區正確
5. 確認每張卡的主要場次與來源場次資訊正確

### 第三輪：異常與邊界

1. 投稿時網路中斷
2. 同篇文章被重複點擊核准
3. 管理員先退回後再重新提交
4. 兩位管理員同時操作同一筆文章
5. 發布匯出途中失敗
6. 已公開文章需要撤回或重匯出

## 14. 暫不處理事項

本版技術決策稿先不處理：

- 自動化 repo 寫檔與自動部署

- 大規模 SEO 細節優化

- 文章排序演算法

---

## 15. 結論

本文件的正式技術結論是：

> 匯集區短期採用「後端審核池 + ready_to_publish + 手動發布匯出 + 前端 published snapshot」的雙階段發佈制。

其核心目的不是先把流程做得最自動，而是先把：

- 發佈責任
- 公開內容來源
- 類型分類
- 測試驗證

這四件事做穩。

後續若真人測試穩定，再評估是否升級成更自動化的發佈流程。
